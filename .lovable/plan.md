

## Plan: Fix Kogan PDF-CSV Pairing to Use Period-Based Matching

### Problem

The current pairing logic tries to match Kogan PDFs to CSVs by extracting a 5+ digit number from the CSV's `settlement_id` and comparing it to the PDF's AP Invoice doc numbers. This almost never works because:

- **PDF doc numbers** are AP Invoice numbers (e.g. `344840`, `360140`) extracted from the PDF text
- **CSV settlement_ids** are generated by the generic CSV parser from whatever column maps to `settlement_id` — typically a date range or order-based ID, NOT an AP Invoice number

Result: only 1 accidental match out of many files. The pairing strategy is fundamentally wrong for Kogan.

### Fix: Period-based pairing

Kogan CSVs and PDFs cover the same **month**. The PDF contains a transfer/payment date and line item dates (e.g. Feb 2026). The CSV has `period_start`/`period_end`. The correct pairing strategy is:

1. Extract the **month/year** from each PDF (from `transferDate` or line item dates already parsed by `parseKoganRemittancePdf`)
2. Extract the **month/year** from each CSV (from `period_start`/`period_end` in the parsed settlement)
3. Match PDF to CSV when they cover the same month

Additionally, use the PDF's AP Invoice doc number as a secondary match: embed the doc number into the CSV's `settlement_id` when a pair is confirmed, so DB lookups also work.
